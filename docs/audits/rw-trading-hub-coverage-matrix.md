## Observations

1. **The product has outgrown its specification.** The script is at `v0.3.87`; the canonical design doc (`docs/scripts/rw-trading-hub.md`) still reads "Active State: v0.1.11, slice 4," scopes inline intel as a future "v0.2.0," and `CONTEXT.md` doesn't list the hub among active scripts at all. The whole pricing brain (PricingEngine, BBEngine, ladders, bands, velocity, classifier), the entire Advertise theming/copy/identity system (#315–#326), and the auction-history DB exist in code with **no current-state doc**. Issue history (commit log + memory files) is the real source of truth; the prose docs lag by ~80 issues.

2. **Two genuinely deep, central subsystems** carry most of the product value: the **pricing brain** (resale estimation, comps, BB floor, ladders — items #10–13) and the **Advertise hub** (items #20–30). Both are well-tested at the pure-function seam (`test-pricing-engine.js`, `test-advconfig.js`, `test-availabilityline.js`). The inline render layer (`InlineRenderer`, ~1300 lines) is the largest untested-at-integration surface.

3. **Inline intel is the headline feature but is gated and discoverable only by habit.** The auction badge renders only after a user manually expands an auction row; the ledger price-check renders only after tapping a row's panel. There is no tab, no count, no "intel is on" affordance beyond two Settings toggles. A new user could miss the most differentiated capability entirely.

4. **Persistence and connection facts in the docs are stale.** The Store-key table omits at least `rwth_intel_settings`, `rwth_collapsed`, `rwth_bb_rate`, `rwth_velocity_log`, and the theme/copy/identity keys. The documented Supabase host differs from the one in the live `@connect` header (`kozewwpyssyzuyksnoqu.supabase.co`).

5. **A known pricing correctness question is live and unresolved** (memory: `project-deduction-anchor-rule`): the auction-headline deduction may double-count ~5% off `ladder.bazaar`. This is product-critical for a tool whose value is "tells you the right max bid," and the fix is explicitly *not* approved — so it's a standing risk, not a closed item.

6. **Error/feedback states are good per-feature but fragmented.** Each subsystem has its own typed states (badge `loading/error/skipped`, `fetchError`, `scanMessage`, `sellMessage`, save status). There is no unified error surface, which is fine for a userscript but worth noting for support/QA.

## Where the codebase diverges from the intended product

1. **Personal-bazaar inline intel was specced as in-scope and never shipped.** `rw-trading-hub.md` §v0.2.0 explicitly names **two** inline surfaces: the auction house *and* personal bazaars ("verdict rendered on the visible listing rows of the loaded page"). Only the auction-house surface exists — `AuctionScanner` early-returns unless `amarket.php`. The bazaar surface was silently dropped with no doc/ADR recording the decision. (The item-market inline widget, by contrast, was correctly documented as out-of-scope and is correctly absent.)

2. **Branding diverged from "NC17" to neutral, by design — but the design doc still describes the old identity as locked.** The doc's entire §Branding (NC17 wordmark, MPAA gag, fixed bazaar/forum titles) was deliberately replaced by the configurable shop-identity system (#316), yet the doc presents the hardcoded brand as final. Anyone reading the doc would expect hardcoded `NC17` copy that no longer exists.

3. **Velocity tracking is built and recorded but barely surfaced.** It's a full subsystem (`VelocityTracker`, persisted `rwth_velocity_log`, per-class baselines) that surfaces as a single line *inside the inline price card* and nowhere on the Ledger/dashboard. For a "trader's workbench" whose ledger already tracks buy→sold timestamps, days-to-clear is a natural ledger metric that currently has no home in the ledger UI.

4. **Diagnostics are console-only.** The "Test weav3r connection" button (the one user-facing health check for the third-party data path the whole pricing brain depends on) reports nothing to the UI — the result lands in the browser console, which most end users will never open.

5. **The documented test gate is disavowed.** `test-rwth.js` still exists and is wired to the seam, but project memory marks it (and `rwth-assets.md`) as stale and non-authoritative. The repo therefore ships a test file that the team is instructed not to trust, with no doc flagging it as deprecated.
